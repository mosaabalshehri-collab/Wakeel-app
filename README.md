# وكيل (Wakeel) — نظام إدارة الوكالات الشرعية

منصة ويب لتسجيل ومتابعة الوكالات الشرعية (Power of Attorney) المسجلة في
منصة "ناجز"، مع تنبيهات تلقائية للوكالات التي قاربت على الانتهاء.

## المشكلة التي يحلها

من يتعامل مع عدة وكالات شرعية (مكاتب عقارية، محامون، مدراء أعمال) يحتاج
طريقة مركزية لتتبع حالتها وتاريخ انتهائها بدل الرجوع لمنصة ناجز يدوياً
في كل مرة. هذا النظام يوفر:

- تسجيل بيانات الوكالة (الموكِّل، الوكيل، النوع، النطاق، التواريخ)
- إرفاق ملف الوكالة (PDF أو صورة) ومراجعته لاحقاً مباشرة من النظام
- تنبيه تلقائي للوكالات التي تنتهي خلال 30 يوماً
- لوحة تحكم بإحصائيات سريعة (سارية / منتهية / ملغاة)
- بحث وفلترة حسب رقم الوكالة، الأسماء، النوع، أو الحالة (بما فيها فلتر
  مخصص للوكالات التي قاربت على الانتهاء)
- سجل تدقيق (audit log) لكل عملية إضافة/تعديل/حذف، مع صفحة عرض مخصصة
- منع تكرار رقم الوكالة لنفس المستخدم (على مستوى قاعدة البيانات والـ API)
- حماية من هجمات التخمين (rate limiting) على تسجيل الدخول وإنشاء الحسابات
- تأكيد البريد الإلكتروني واسترجاع كلمة المرور عبر روابط بريدية مؤقتة
- وضع داكن (Dark Mode) يتبع إعداد النظام تلقائياً مع إمكانية التبديل
  اليدوي، والتفضيل محفوظ بين الزيارات
- حذف آمن (Soft Delete): الوكالة المحذوفة تُنقل لسلة محذوفات لمدة 30
  يوماً قبل حذفها نهائياً، مع إمكانية الاستعادة في أي وقت خلال هذه المدة
- نسخ احتياطي دوري لقاعدة البيانات مع تنظيف تلقائي للنسخ القديمة
- إشعارات فورية (Toast) لنتيجة كل عملية، وتحذير قبل مغادرة فورم فيه
  تعديلات غير محفوظة

## الستاك التقني

| الطبقة | التقنية | السبب |
|---|---|---|
| Frontend | Next.js 15 (App Router) + TypeScript + Tailwind CSS | مشروع واحد متكامل، سهل النشر، شائع في سوق العمل |
| Backend | Next.js API Routes | لا حاجة لسيرفر منفصل |
| قاعدة البيانات | SQLite عبر `node:sqlite` (مدمجة في Node.js 22+) | صفر اعتماديات native، تعمل فوراً على أي جهاز |
| المصادقة | JWT (jsonwebtoken) + bcrypt | بسيطة وموثوقة لمشروع بهذا الحجم |
| التحقق من المدخلات | Zod | فحص صارم لكل بيانات الفورم في الـ API |

### لماذا SQLite بدل PostgreSQL؟

القرار الأصلي كان PostgreSQL + Prisma، لكن أثناء التطوير في بيئة بقيود
شبكة (sandboxed)، تعذّر تحميل محركات Prisma الثنائية. بدل تعليق العمل،
تم الانتقال إلى `node:sqlite` — وحدة SQL حقيقية ومدمجة بالكامل في
Node.js 22+، بدون أي تثبيت خارجي.

كل التعامل مع قاعدة البيانات معزول في `src/lib/repositories/` (نمط
Repository)، لذلك الترقية لاحقاً إلى PostgreSQL (موصى به فعلياً عند
النشر الحقيقي على Vercel) تتطلب فقط إعادة كتابة هذه الملفات الثلاثة
دون التأثير على أي API route أو واجهة مستخدم. راجع قسم "الترقية إلى
PostgreSQL" أدناه.

## البريد الإلكتروني (تأكيد الحساب واسترجاع كلمة المرور)

المشروع قيد البناء حالياً بدون خدمة بريد فعلية مربوطة. `src/lib/mailer.ts`
مبني بحيث يعمل فوراً بمجرد إضافة `RESEND_API_KEY` في `.env.local` (راجع
`.env.example`) — لا حاجة لتعديل أي كود آخر.

**في وضع التطوير الحالي** (بدون المفتاح)، أي رسالة بريد (تأكيد حساب أو
استرجاع كلمة مرور) تُطبع كاملة في الطرفية بدل إرسالها فعلياً، بما في ذلك
الرابط الذي يحتوي التوكن. هذا يسمح باختبار التدفقين كاملين محلياً:

```bash
npm run dev
# سجّل حساباً جديداً، أو اطلب استرجاع كلمة مرور
# راقب الطرفية: ستجد رابطاً مثل
# http://localhost:3000/verify-email?token=...
# انسخه والصقه في المتصفح لإكمال التدفق
```

## رفع المرفقات

ملفات الوكالات (PDF أو صور JPG/PNG، حتى 5 ميجابايت) تُخزَّن كملفات فعلية
في `data/uploads/` بأسماء عشوائية تماماً (غير مرتبطة باسم الملف الأصلي أو
رقم الوكالة)، وتُقدَّم فقط عبر `/api/poa/[id]/attachment` بعد التحقق من
أن طالب الملف هو مالك الوكالة فعلاً — الملفات ليست في `public/` ولا يمكن
الوصول لها مباشرة بمعرفة المسار.

## الاختبارات

منطق الأعمال الأساسي (في `src/lib/repositories/` و`src/lib/auth.ts`)
مغطّى باختبارات وحدة باستخدام Node's test runner المدمج (بدون أي
مكتبة اختبار خارجية):

```bash
npm test
```

كل اختبار لمنطق الوكالات يعمل على قاعدة بيانات SQLite منفصلة في
الذاكرة (`:memory:`)، عبر `createPoaRepository` — دالة factory تأخذ
اتصال قاعدة بيانات بدل الاعتماد على singleton عام، فتسمح بحقن اتصال
اختباري معزول تماماً عن بيانات التطوير الفعلية.

من ضمن الاختبارات: حالة اختبار تراجع (regression test) تغطي خطأً
حقيقياً تم اكتشافه أثناء التطوير، حيث كان التعديل الجزئي (PATCH)
يمسح حقولاً موجودة لم تُرسل في الطلب.

اختبارات `src/lib/validation/__tests__/` تغطي خطأً آخر تم اكتشافه:
عدم وجود أي تحقق يمنع تاريخ انتهاء الوكالة من أن يكون قبل تاريخ
إصدارها. الإصلاح يعمل على مستويين — تحقق أولي في Zod schema (يكتشف
الحالة الواضحة عند إرسال الحقلين معاً)، وتحقق إضافي في API route
نفسه يقارن القيم المُرسلة مع البيانات المخزّنة فعلاً، لتغطية حالة
PATCH الجزئي حيث يُرسل تاريخ واحد فقط دون الآخر.

## التشغيل محلياً

```bash
npm install
npm run dev
```

افتح `http://localhost:3000` — ستُحوَّل تلقائياً لصفحة تسجيل الدخول.
قاعدة البيانات (`data/wakeel.db`) تُنشأ تلقائياً عند أول تشغيل.

اختياري: انسخ `.env.example` إلى `.env.local` وغيّر `JWT_SECRET` لقيمة
عشوائية (مطلوب فعلياً فقط عند النشر الحقيقي).

## الترقية إلى PostgreSQL (موصى بها قبل النشر النهائي)

1. ثبّت Prisma: `npm install prisma @prisma/client` ثم `npx prisma init`
2. حوّل تعريفات الجداول في `src/lib/db.ts` إلى `schema.prisma`
3. أعد كتابة الدوال في كل ملف داخل `src/lib/repositories/` لتستخدم
   `prisma.poweOfAttorney.findMany()` بدل استعلامات SQL الخام — التوقيعات
   (function signatures) يمكن أن تبقى كما هي تماماً
4. لا حاجة لتغيير أي شيء في `src/app/api/` أو `src/components/`

## الهوية البصرية

التصميم مستوحى من منصة "ناجز" الرسمية (نفس الجهة التي تصدر الوكالات
التي يديرها هذا النظام)، عشان يعطي إحساساً مألوفاً وموثوقاً للمستخدم:

- **الألوان**: أخضر زاهي رسمي (`#00754A`) كلون أساسي، مع لمسة ذهبية
  (`#C9A24B`) للتمييز، على خلفية بيضاء/رمادي فاتح نظيفة
- **الشعار**: علامة صح بسيطة داخل مربع أخضر مع نقطة ذهبية، بنفس روح
  شعارات "ناجز" التي تجمع بين رمز التحقق واسم المنصة (`src/components/WakeelLogo.tsx`)
- **الخلفية**: نقاط زخرفية خفيفة جداً على صفحات الدخول والتسجيل
  (class `.dotted-bg` في `globals.css`)، نفس الأسلوب الموجود في
  الصفحة الرئيسية لناجز
- متغيرات الألوان كلها مركزية في `src/app/globals.css` (`:root`)،
  فتغيير الهوية البصرية لاحقاً يكون من مكان واحد فقط
- **عناصر زخرفية**: دوائر متحدة المركز وخطوط شعاعية ذهبية بأسلوب
  ختم رسمي (`src/components/SealPattern.tsx`)، تظهر بجانبي صفحات
  الدخول والتسجيل لتعبئة الفراغ بدون تشتيت الانتباه عن الفورم

## الوضع الداكن (Dark Mode)

يتبع إعداد النظام تلقائياً (`prefers-color-scheme`) عند أول زيارة، مع
زر تبديل يدوي في الهيدر (وفي صفحات الدخول/التسجيل) يتجاوز إعداد النظام
ويُحفظ في `localStorage` — التفضيل يبقى ثابتاً بين الزيارات حتى لو
تغيّر إعداد النظام لاحقاً.

كل الألوان في المشروع معرّفة كمتغيرات CSS (`var(--color-*)`) في
`globals.css`، بنسختين: نسخة افتراضية (`:root`) ونسخة الوضع الداكن
(`.dark`). بما أن كل الكود — سواء inline styles أو Tailwind classes —
يستخدم نفس المتغيرات، التبديل بين الوضعين لا يحتاج أي تعديل في
المكونات أو الصفحات؛ فقط في القيم المعرّفة مرة واحدة في `globals.css`.

**ملاحظة تقنية مهمة**: السكربت الذي يطبّق الوضع الصحيح قبل أول رسم
للصفحة (لتفادي وميض المحتوى الخاطئ FOUC) محقون عبر `next/script`
بإستراتيجية `beforeInteractive` في `layout.tsx` — استخدام وسم
`<script>` خام مباشرة داخل مكوّن React غير موصى به ويُصدر تحذيراً
صريحاً، فضلاً عن أنه لا يُعاد تنفيذه عند التنقل بين الصفحات عبر
client-side navigation (بخلاف `beforeInteractive` الذي يضمن التنفيذ
في كل تحميل كامل للصفحة).

## الحذف الآمن (Soft Delete) وسلة المحذوفات

حذف وكالة من `/poa` لا يحذفها فعلياً من قاعدة البيانات؛ بدلاً من ذلك
يُعيَّن عمود `deleted_at`، وتختفي الوكالة من القوائم النشطة فوراً مع
بقائها متاحة في `/poa/trash` لمدة `TRASH_RETENTION_DAYS` (30 يوماً
افتراضياً، معرّفة في `createPoaRepository.ts`) قبل أن تصبح مؤهلة
للحذف النهائي.

**نقطة تصميم مهمة**: القيد الذي يمنع تكرار رقم الوكالة لنفس المستخدم
(`idx_poa_unique_number_per_user` في `db.ts`) هو **فهرس فريد جزئي**
(`UNIQUE INDEX ... WHERE deleted_at IS NULL`) وليس قيد `UNIQUE` عادياً
على العمود. الفرق جوهري: لو استُخدم قيد عادي، حذف وكالة ناعماً كان
سيمنع نهائياً إعادة استخدام رقمها لاحقاً، لأن السجل القديم يبقى موجوداً
فعلياً في قاعدة البيانات ويصطدم بالقيد. الفهرس الجزئي يستثني الصفوف
المحذوفة من فحص التفرّد، فرقم الوكالة المحذوفة متاح لإعادة الاستخدام
فوراً.

نفس المنطق ينطبق عند الاستعادة: لو أنشأ المستخدم وكالة جديدة بنفس رقم
وكالة محذوفة موجودة في السلة، ثم حاول استعادة القديمة، الاستعادة تُرفض
(409) برسالة توضح ضرورة تعديل رقم إحداهما أولاً — `restore()` في
`createPoaRepository.ts` يتحقق من هذا التعارض صراحة قبل الاستعادة.

حذف وكالة نهائياً من سلة المحذوفات (`DELETE /api/poa/trash/[id]`)
يحذف أيضاً الملف المرفق الفعلي من `data/uploads/` إن وجد.

## النسخ الاحتياطي لقاعدة البيانات

بما أن المشروع يستخدم SQLite بملف واحد (`data/wakeel.db`)، فقدان هذا
الملف (تلف قرص، خطأ بشري، فشل خادم) يعني فقدان كل بيانات كل المستخدمين
نهائياً بدون أي طريقة استرجاع — لا توجد آلية نسخ احتياطي تلقائية من
قاعدة البيانات نفسها كما في الخوادم المُدارة. `src/lib/backup.ts` يوفر
الحد الأدنى الضروري من الحماية ضد هذا السيناريو.

**الآلية**: أمر SQL المدمج `VACUUM INTO` (وليس نسخ الملف مباشرة) —
ينتج لقطة (snapshot) كاملة ومتسقة من قاعدة البيانات حتى مع وجود عملية
كتابة جارية وقت أخذ النسخة، بخلاف نسخ الملف الخام المعرّض لإنتاج نسخة
تالفة في تلك الحالة. النسخ تُحفظ في `data/backups/` (خارج Git)، مع
الاحتفاظ بآخر 14 نسخة فقط تلقائياً (`BACKUP_RETENTION_COUNT` في
`backup.ts`) لتفادي نمو غير محدود لحجم القرص.

**التفعيل**: نقطة النهاية `/api/admin/backup` (POST لتشغيل نسخة فورية،
GET لعرض القائمة) محمية بمفتاح سري منفصل تماماً عن مصادقة المستخدمين
العاديين (`BACKUP_SECRET` في `.env.local`، يُمرَّر عبر رأس
`Authorization: Bearer <secret>`). إن لم يُضبط هذا المتغير، نقطة
النهاية ترفض كل الطلبات افتراضياً (الأمان الافتراضي "مرفوض" وليس
"مسموح"). لا توجد جدولة تلقائية داخل كود التطبيق نفسه — يجب ربط هذه
النقطة باستدعاء دوري (يومي مثلاً) من خدمة جدولة خارجية (cron job على
الخادم، GitHub Actions scheduled workflow، أو خدمة مثل
cron-job.org)، لأن Next.js في وضع الخادم العادي لا يوفر آلية cron
مدمجة موثوقة.

## تحسينات تجربة الاستخدام

- **إشعارات فورية (Toast)**: كل عملية حفظ، حذف، استعادة، أو خطأ تُظهر
  إشعاراً مرئياً أعلى الصفحة (`src/components/ToastProvider.tsx`) بدل
  الاعتماد فقط على تغيّر الصفحة كدليل ضمني على النجاح.
- **تحذير عند مغادرة فورم فيه تعديلات غير محفوظة**: زر "إلغاء" في
  فورم الوكالة يطلب تأكيداً صريحاً إن كانت هناك تعديلات لم تُحفظ، وكذلك
  محاولة إغلاق التبويب أو تحديث الصفحة (`useUnsavedChangesWarning` في
  `src/lib/hooks/`). هذا الحذف يغطي فقط مغادرة الموقع بالكامل (عبر
  `beforeunload`، مدعوم بشكل موثوق في كل المتصفحات) — لا يغطي التنقل
  الداخلي بين صفحات Next.js تلقائياً، لأن App Router لا يوفر حتى الآن
  واجهة رسمية لاعتراض التنقل الداخلي؛ الأزرار الصريحة (مثل "إلغاء")
  تتحقق من حالة التعديلات يدوياً.

## ملاحظات أمنية قبل أي نشر فعلي

- غيّر `JWT_SECRET` لقيمة عشوائية قوية (لا تستخدم القيمة الافتراضية)
- فعّل HTTPS (الكوكي معدّة على `secure: true` تلقائياً في وضع الإنتاج)

### حماية المسارات (withAuth)

كل API route يحتاج مستخدماً مسجلاً دخوله مُغلَّف بـ `withAuth` من
`src/lib/withAuth.ts` بدل تكرار فحص المصادقة يدوياً في كل دالة. هذا
يضمن بنيوياً أن أي route جديد يُكتب بهذا النمط لا يمكن أن ينسى فحص
المصادقة — الفحص يحدث قبل وصول الكود لمنطق العمل دائماً، وليس خطوة
يعتمد تطبيقها على تذكّر المطوّر لها.

### Rate limiting (مطبّق فعلياً)

`/api/auth/login` (10 محاولات / 5 دقائق) و`/api/auth/register`
(5 محاولات / 15 دقيقة) محميان بـ rate limiter بسيط في الذاكرة
(`src/lib/rateLimit.ts`)، مفتاحه عنوان IP. هذا كافٍ لخادم واحد
(single instance) — وهو وضع هذا المشروع حالياً.

**قيد معروف:** الحد بالذاكرة، فيُعاد تصفيره عند إعادة تشغيل السيرفر،
ولا يُشارك بين عدة نسخ من التطبيق (multi-instance/serverless). عند
النشر على بنية موزّعة، استبدله بحل مشترك مثل Redis.

ملاحظة عند الاختبار اليدوي: لأن العدّاد مرتبط بالـ IP ومستمر طوال
النافذة الزمنية، تجربة عدة طلبات متتالية بنفس الجلسة (مثلاً عبر
curl بنفس الجهاز) قد تصل للحد المسموح أسرع مما تتوقع إذا كانت هناك
محاولات سابقة لم تنتهِ نافذتها الزمنية بعد — هذا هو السلوك الصحيح
المقصود، وليس خللاً.

### رفع الملفات

`/api/poa/upload` يتحقق من نوع الملف (PDF/JPG/PNG فقط عبر MIME type)
وحجمه (5 ميجابايت كحد أقصى) قبل القبول، ويولّد اسم ملف عشوائي تماماً
بدل استخدام اسم الملف الأصلي — يمنع هذا أي محاولة path traversal أو
الكتابة فوق ملفات موجودة. `/api/poa/[id]/attachment` يتحقق من ملكية
الوكالة قبل تقديم الملف، فالمسار العشوائي وحده غير كافٍ للوصول.
